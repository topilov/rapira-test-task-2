# Тестовое задание. Часть 2 — Order Reservation, Cancel, Idempotency

## Контекст

Есть упрощённый backend криптовалютной биржи.

Сейчас система умеет создавать **LIMIT BUY** ордера, но делает это некорректно:
- не резервирует средства;
- не поддерживает идемпотентность клиентского запроса;
- не умеет корректно отменять ордер с освобождением резерва.

## Что нужно реализовать

### 1. Создание LIMIT BUY ордера

При создании ордера система должна:

- принять `clientOrderId`, `userId`, `symbol`, `price`, `quantity`;
- проверить входные данные;
- посчитать сумму ордера: `price * quantity`;
- проверить достаточность `availableUsdt`;
- уменьшить `availableUsdt`;
- увеличить `reservedUsdt`;
- создать ордер в статусе `NEW`.

### 2. Идемпотентность

Если для одного пользователя повторно приходит запрос с тем же `clientOrderId`, система не должна:
- создавать новый ордер;
- повторно резервировать средства.

Она должна вернуть уже существующий ордер.

### 3. Отмена ордера

Нужно реализовать отмену ордера:
- отменять можно только ордер в статусе `NEW`;
- при отмене ордера сумма резерва должна быть освобождена:
  - `reservedUsdt` уменьшается;
  - `availableUsdt` увеличивается;
- ордер переходит в статус `CANCELLED`.

### 4. Ошибки

Нужно корректно обработать:
- недостаточно средств;
- повторную отмену;
- отмену несуществующего ордера;
- попытку создать ордер с невалидными параметрами.

## Упрощения

В рамках задачи:
- поддерживаются только **LIMIT BUY** ордера;
- комиссия не учитывается;
- исполнение ордера не требуется;
- matching engine не нужен;
- обработка конкурентного доступа не требуется.
