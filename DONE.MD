Реализовано:
- создание `LIMIT BUY` с резервированием `availableUsdt -> reservedUsdt`
- идемпотентность по `(userId, clientOrderId)`: повтор с тем же payload возвращает существующий ордер, с другим payload даёт конфликт
- отмена только для `NEW` с освобождением резерва и переводом в `CANCELLED`
- HTTP-обработка ошибок: `400` для невалидного запроса, `404` для отсутствующего ордера, `409` для конфликтов состояния/баланса/идемпотентности
- тесты сервиса и API

Инварианты:
- сумма ордера считается как `price * quantity`
- повторный idempotent create не создаёт новый ордер и не резервирует деньги повторно
- после отмены весь резерв по ордеру возвращается в `availableUsdt`

Production:
- единая транзакция для обновления ордера и баланса
- персистентные репозитории вместо in-memory
- bean validation и обработка malformed JSON / binding errors
- аудит и structured logging
